## A 版 MVP — Web 信用卡參考版

### 核心原理
利用信用卡的標準尺寸（**85.6mm × 54.0mm**）作為比例參考，透過影像分析計算魚的實際長度。

### 使用流程
1. 釣到魚，放置於平坦地面
2. **將信用卡放在魚旁邊（同一平面）**
3. 拍照（確保信用卡和魚完整入鏡）
4. AI 自動偵測信用卡邊界
5. AI 自動偵測魚頭到尾鰭的長度（pixel 距離）
6. 透過信用卡 pixel 尺寸換算實際 cm
7. 顯示結果並記錄環境資訊

### 功能規格
| 功能 | 說明 | 優先級 |
|------|------|--------|
| 信用卡偵測 | 自動辨識照片中的信用卡邊界 | P0 |
| 信用卡尺寸校正 | 以 85.6mm 為基準計算 pixel-to-cm 比例 | P0 |
| 魚體偵測 | 自動框選魚輪廓 | P0 |
| 頭尾定位 | 標記魚頭前端與尾鰭末端 | P0 |
| 長度計算 | 計算魚的直線長度（含曲線處理 P2） | P0 |
| 手動調整 | 允許用戶微調頭尾位置 | P1 |
| 拍照引導 | 提示用戶將信用卡與魚放同一平面 | P1 |
| 角度校正 | 偵測拍攝角度偏差並提示 | P2 |

### 技術實作
- **前端**：React / Vue 網頁應用
- **相機存取**：MediaDevices API
- **圖片處理**：Canvas API
- **信用卡偵測**：YOLO / OpenCV 邊緣偵測 / 模板匹配
- **魚體偵測**：SAM (Segment Anything Model) 或 YOLO segmentation
- **比例換算**：信用卡 pixel 寬度 → 換算因子 → 魚 pixel 長度 → 實際長度
- **誤差處理**：透視變形校正（假設魚與卡在同一平面）
- **資料儲存**：IndexedDB / LocalStorage

### 預期精準度
| 條件 | 預期誤差 |
|------|----------|
| 理想條件（正上方、同一平面、光線充足）| ±2-5% |
| 一般條件（些許角度偏差）| ±5-10% |
| 較差條件（大角度、彎曲魚身）| ±10-20% |

### 限制與假設
- 信用卡與魚必須在同一平面上
- 建議相機盡量垂直於地面拍攝（減少透視誤差）
- 魚身彎曲時，直線長度會低估實際長度

## B 版 MVP — iOS App（LiDAR / ARKit 純手機技術版）

### 核心原理
利用 iPhone 的 LiDAR 感應器、ARKit 框架與雙鏡頭深度資訊，在不放置參考物的情況下，直接透過空間深度推估魚的實際長度。

### 適用裝置
| 裝置類型 | 支援技術 | 說明 |
|----------|----------|------|
| iPhone 12 Pro/Pro Max 之後 Pro 系列 | LiDAR + ARKit + 雙鏡頭 | 完整功能，精準度最高 |
| iPhone X 之後（有雙鏡頭）| ARKit + 雙鏡頭深度 | 無 LiDAR，精準度較低 |
| iPhone SE / 單鏡頭機型 | ARKit（僅平面偵測）| 精準度最低，不建議 |

### 使用流程
1. 釣到魚，放置於平坦地面
2. **直接開啟 App，將相機對準魚**
3. ARKit 偵測地面平面
4. LiDAR / 深度感應器掃描魚體立體輪廓
5. 用戶點擊魚頭與尾鰭位置（或 AI 自動偵測）
6. ARKit 計算兩點在 3D 空間中的實際距離
7. 顯示結果並記錄環境資訊

### 功能規格
| 功能 | 說明 | 優先級 |
|------|------|--------|
| AR 相機啟動 | 開啟相機並初始化 ARKit Session | P0 |
| 平面偵測 | ARKit 自動偵測地面平面 | P0 |
| 深度感知 | LiDAR / TrueDepth 取得場景深度圖 | P0 |
| 頭尾定位 | 用戶在螢幕上點擊魚頭與尾鰭（3D 空間位置） | P0 |
| 空間距離計算 | ARKit 計算兩點在世界座標中的實際距離 | P0 |
| 魚體偵測 | AI 自動試圖標記魚輪廓（輔助用） | P1 |
| 視覺指引 | AR 線條即時顯示測量路徑與結果 | P1 |
| 遮擋處理 | 處理魚身被草/石頭部分遮擋的情況 | P2 |

### 技術實作
- **語言**：Swift
- **框架**：ARKit + SceneKit / RealityKit
- **深度來源**：LiDAR Scene Depth（A12Z 後支援）/ TrueDepth / 雙鏡頭視差
- **平面偵測**：ARKit Plane Detection（水平平面）
- **距離計算**：simd_distance(worldPosition1, worldPosition2)
- **魚體偵測**：可選接入 CoreML 模型輔助標記頭尾
- **天氣 API**：OpenWeatherMap
- **資料儲存**：Core Data / UserDefaults（本地），iCloud / Firebase（雲端第二階段）

### 預期精準度
| 裝置 / 條件 | 預期誤差 | 說明 |
|-------------|----------|------|
| LiDAR 機型 + 理想條件 | ±5-10% | 光線充足、平面清晰 |
| LiDAR 機型 + 一般戶外 | ±10-15% | 草地、沙地、反光水面 |
| 雙鏡頭深度 + 理想條件 | ±15-20% | 無 LiDAR 機型 |
| 魚身懸空 / 彎曲 | ±15-25% | 尾巴翹起或身體彎曲時 |

### 限制與風險
- **僅限 iPhone Pro 系列**：LiDAR 精準度最好，但非全部用戶都有
- **光線需求**：過暗或過亮（水面反光）會影響 LiDAR / 雙鏡頭深度
- **平面品質**：沙地、草地、碎石地的平面偵測不穩定
- **魚的姿態**：魚身彎曲或尾巴翹起時，直線距離會低估實際長度
- **水漬反光**：濕滑的魚身可能造成深度誤判

## A/B 版比較總結

| 比較項目 | A 版 Web（信用卡參考） | B 版 iOS App（LiDAR/ARKit） |
|----------|------------------------|----------------------------|
| **平台** | Web（跨平台） | iOS App（需下載安裝） |
| **使用者操作** | 需放信用卡在魚旁 | 直接拍，無需參考物 |
| **硬體需求** | 任何手機 + 信用卡 | 需 iPhone Pro 系列（LiDAR）或雙鏡頭機型 |
| **預期精準度** | ±2-10% | ±5-20%（看裝置與環境） |
| **技術複雜度** | 中等 | 高（ARKit + Swift） |
| **開發成本** | 較低（Web 技術） | 較高（原生 App + 蘋果開發者帳號） |
| **開發者需求** | 前端工程師 | iOS 工程師 |
| **MVP 目標** | 驗證參考物方案的準確度 | 驗證無參考物 AR 方案的可行性 |

## 開發順序建議

### 第一階段（2-3 週）：A 版 Web 核心
1. 照片上傳與預覽功能
2. 信用卡偵測（簡單版本：矩形偵測）
3. 手動標記魚頭/魚尾（先不做 AI 自動偵測）
4. 長度計算（pixel 比例換算）
5. 基本結果顯示

### 第二階段（2-3 週）：A 版優化
6. AI 自動魚體偵測
7. AI 自動頭尾定位
8. GPS / 時間 / 天氣自動記錄
9. 資訊卡片設計與分享功能

### 第三階段（3-4 週）：B 版 iOS App
10. ARKit 基礎框架搭建
11. LiDAR 深度感知整合
12. 地面平面偵測
13. 點擊測量頭尾距離
14. 視覺指引與結果呈現

### 第四階段（2 週）：A/B 版並行測試
15. 訂定量測準確度驗證方法（與實際尺量測比對）
16. 收集兩版各 50-100 筆量測數據
17. 分析誤差分佈與影響因子
18. 決定主力開發方向

### 第五階段（後續）
19. 根據 A/B 版數據決定主力方向
20. 歷史記錄與統計功能
21. 社群分享與用戶系統
22. 考慮開發 Android 版

## 風險與假設

| 風險 | 影響 | 因應方案 |
|------|------|----------|
| A 版準確度不如預期 | 產品核心價值受損 | 收集數據，優化模型；或改進拍照引導 |
| B 版 iOS App 開發複雜度高 | 時程延誤 | 可先以雙鏡頭深度版本驗證，後續再加入 LiDAR |
| B 版僅限 iPhone Pro 用戶 | 市場覆蓋率低 | 評估是否值得投資，或轉向 A 版為主 |
| 天氣 API 限制 | 環境記錄不完整 | 多備援 API 來源 |
| 信用卡全球尺寸不同 | 參考物不統一 | 支援多國信用卡尺寸設定 |
| Android 用戶無法使用 B 版 | 部分市場無法覆蓋 | MVP 先聚焦 iOS，後續評估 ARCore 版本 |
